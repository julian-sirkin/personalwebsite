Recipe Tracker

Technologies used

Project Overview

The final (full stack) application uses Mongo, Express, and Node on the backend with Javascript and HTML on the front end. I am currently rebuilding the front end using React. The project uses the ‘edamam’ API (https://developer.edamam.com/) and allows the user to seach for recipes, save favorites to a list, and to remove recipes from the list. The project includes the start of an interaction with the Amazon Fresh API. Their API is not fully functional yet. When it is, the application will be enhanced to create a shopping list from the selected recipe(s), and will also be able to order the groceries from Amazon.

What I learned

The primary lesson I learned from this project was prioritization. My plan was to use the MERN stack. However, due to the very tight project deadline I did not have the time to add React in version 1. As such, I focused on building a more complicated back end (with an HTML and jQuery front end). I am now (after the class) rebuilding the front end using React.
While working through the back end code I learned how to call on a third party API and how to incorporate and parse the query results. This way I am able to use the minimal amount of information (recipe IDs) so I can bundle and query a large amount of information at once (from edamam). This allows me to process the information within my application more efficiently.

Bucket List

Technologies used

Project Overview

This project was designed to create a bucket list using: Mongo, Express, and Mongoose (for the back end) and Javascript, HTML, and CSS (for the front end). My group decided to create a highly-polished app that would exactly meet the requirements we were given. Users are able to add items to the bucket list, view details of items, delete bucket list items, and mark the items as completed. While this is a simple application, I feel that it really helped me understand project scope, and how to balance the scope with the project’s limited timeframe.

What I learned

This project really helped me learn a lot about team programming Best practices. I am used to working in a kitchen “brigade”, but the dynamics of software teams are quite different, and carry their own challenges. It was different, but truly exciting. The most important aspect, I learned, was clear communication and a good division of labor.

Workout Tracker

Technologies used

Project Overview

This was my first full-stack application. It is used to track people’s workouts. In addition, users can add different exercises to their workouts. The backend was built with Ruby on Rails, with a Javascript front end. The backend allows users to view not only their workouts, but also any exercises created by any other users. Workouts and exercises are joined using additional information (such as reps and sets) to allow for individualization of workouts.

What I learned

This project was really useful for several reasons. I became familiar with Ruby on Rails. I was able to take the prompt (CRUD on a single resource) and expand it so I could become more familiar with the framework and concepts. I learned many new things from this, such as destroying dependents, based on relationships in join tables. I also learned something important about myself: When I have extra time, I gravitate towards additional functionality.

Tic Tac Toe

Technologies used

Project Overview

A single-page application. This was the first application that I ever built. The point of the project is to learn how to interact with an API (provided) With Ajax calls. Then I had to parse and use the information returned from the server. The project contains a custom game engine and a computer player that can predict the next proper move.

Learning Process

In this project I learned the importance of planning and flexibility. I first built the game engine. It was capable of initializing a game and detecting victory conditions. As I began working with the API, I realized that I had to refactor the game engine. I learned the importance of being willing to apart key (already working) sections of the code, and reworking it to operate with larger future goals.
